# 펜플로우 (PenFlow) — 마커·프롬프트 규약서

> 기준 문서: `02-functional-design.md`(2·8·9·10·11·15·16장, R-2.x·R-3.x·R-7.x·R-8.x), `03-implementation-checklist.md`(0·3·5·6장)
> 목적: 본문 안에 삽입 지점을 표시하는 **마커**의 문법과, 본문을 만들어내는 **프롬프트**의 형식을 **한 곳에 확정**한다.
> 왜 먼저인가: 생성기(③)·비주얼(⑨)·합성기(④)·검증기(⑩)가 **모두 이 규약 하나에 의존**한다. 여기가 흔들리면 하위 컴포넌트가 전부 깨진다. (03의 "마커 규약 확정 — 1곳에 정의" 항목)

---

## 1. 설계 원칙

1. **단일 출처(Single Source of Truth)**: 마커 문자열·정규식·타입은 **상수 파일 1개**(`markers.ts`)에만 정의하고, 모든 컴포넌트가 이를 import 한다. (03 DoD)
2. **변환 안전(Round-trip Safe)**: 마커는 HTML↔Markdown 변환을 왕복해도 **1개도 손실되지 않아야** 한다(R-8.4). 그래서 HTML 주석(`<!-- -->`)이나 특수 태그가 아니라, **양 포맷 모두에서 일반 텍스트로 통과되는 플레인 토큰**을 쓴다.
3. **순서 보존**: 본문 내 마커 등장 순서가 곧 삽입 순서다. 임의 재배치 금지(R-3.2).
4. **결정적 파싱**: 정규식 1개로 모든 마커를 한 번에 스캔할 수 있어야 한다.

---

## 2. 마커 문법

### 2.1 기본형

```
[[PF:<TYPE>:<KEY>]]
```

- `[[PF:` … `]]` 로 감싸 일반 본문 텍스트와 충돌하지 않게 한다. (`PF` = PenFlow)
- `<TYPE>` — 대문자 토큰(아래 표)
- `<KEY>` — 식별자. 순번(`1`,`2`,…) 또는 리소스 키. 생략형은 `0`.
- 공백 없음. 한 줄에 단독으로 두는 것을 원칙으로 한다(블록 요소가 대부분이라).

**전체 스캔 정규식(참조용)**

```
/\[\[PF:([A-Z0-9]+):([A-Za-z0-9_-]+)\]\]/g
```

### 2.2 마커 종류

| TYPE | 의미 | 대응 리소스 | KEY 규칙 | 리소스 없을 때 |
|---|---|---|---|---|
| `H1THUMB` | 대표(H1) 썸네일 지점 | Visual(대표) | `0` 고정(1개) | 스킵 |
| `H2THUMB` | 소제목(H2) 썸네일 지점 | Visual(소제목) | 소제목 순번 `1`,`2`,… | 스킵 |
| `IMG` | 본문 이미지 지점 | Visual(본문) | 순번 `1`,`2`,… | 스킵 |
| `PRODUCT` | 상품표 | 상품 데이터 | 상품 키 | 스킵 |
| `SHOP` | 쇼핑/제휴 링크 | 제휴 링크 | 링크 키 | 스킵 |
| `BACKLINK` | 백링크 블록 | 백링크 데이터 | 키 | 스킵 |
| `CTA` | CTA 버튼 | 버튼 정의 | 키 | 스킵 |
| `SOURCE` | 출처 링크 | 출처 목록 | 키 | 스킵 |
| `AD` | 광고/협찬 문구 | 문구 텍스트 | `0` 또는 키 | **스킵 금지**(아래 2.4) |

> 소제목(H2) 자체는 마커가 아니라 일반 `## 제목` / `<h2>`로 둔다. **H2 썸네일을 붙일 지점만** `[[PF:H2THUMB:n]]`로 표시한다.

### 2.3 배치 규칙

- `H2THUMB:n`은 n번째 `<h2>` **바로 아래**에 온다.
- 마커는 본문 흐름 상 **삽입될 정확한 위치**에 둔다(합성기가 위치를 옮기지 않는다, R-3.2).
- 동일 TYPE의 KEY는 본문 내에서 **오름차순·연속**이어야 한다(`IMG:1`, `IMG:2`… 건너뜀 금지).

### 2.4 광고/협찬 문구(`AD`)의 특례

- 광고/협찬 옵션이 **ON**이면 `AD` 마커는 **반드시 1개 이상 존재**하고, 대응 문구가 없어도 **스킵하지 않는다**(R-3.4). 누락 시 합성 차단 + 통지.
- 다른 모든 마커는 대응 리소스가 없으면 **조용히 스킵**한다(R-3.1).

### 2.5 02 데이터 모델과의 대응 (용어 매핑)

06의 마커 토큰은 02 15장의 `InsertTask.type`·`Visual.role`과 **같은 개념을 다른 이름**으로 부른다. 코드에서 둘을 오갈 때 아래 표를 단일 기준으로 삼는다. (마커 = 본문 위치 표기 / `InsertTask.type` = 삽입 작업 종류 / `Visual.role` = 이미지 역할)

| 06 마커 TYPE | 02 `InsertTask.type` | 02 `Visual.role` | 비고 |
|---|---|---|---|
| `H1THUMB` | `image` | `H1_THUMB` | 대표 썸네일 |
| `H2THUMB` | `image` | `H2_THUMB` | 소제목 썸네일 |
| `IMG` | `image` | `BODY_IMAGE` | 본문 이미지 |
| `PRODUCT` | `productTable` | — | 상품표 |
| `SHOP` | `shoppingLink` | — | 쇼핑/제휴 링크 |
| `BACKLINK` | `backlink` | — | 백링크 블록 |
| `CTA` | `ctaButton` | — | CTA 버튼 |
| `AD` | `adNotice` | — | 광고/협찬 문구 |
| `SOURCE` | *(02 모델에 없음)* | — | **확장**: 02는 출처를 `Payload.options.includeSourceLink: bool`(전역 on/off)로 두지만, 06은 위치 지정 마커로 확장함(8장 열린 항목) |

> 표기 통일 원칙: 마커는 06 토큰(`IMG`/`SHOP`/…)을, 데이터 모델 필드는 02 이름(`image`/`shoppingLink`/…)을 그대로 쓴다. **변환은 이 표 한 곳에서만** 한다(`markers.ts`에 매핑 상수로 둔다).

## 3. 마커 상수 파일 (`markers.ts`) — 권장 골격

```ts
// 펜플로우 마커 단일 출처. 모든 컴포넌트는 여기만 import 한다.
export const MARKER_TYPES = [
  'H1THUMB', 'H2THUMB', 'IMG',
  'PRODUCT', 'SHOP', 'BACKLINK', 'CTA', 'SOURCE', 'AD',
] as const;
export type MarkerType = typeof MARKER_TYPES[number];

export const MARKER_RE = /\[\[PF:([A-Z0-9]+):([A-Za-z0-9_-]+)\]\]/g;

export interface Marker {
  type: MarkerType;
  key: string;
  index: number;   // 본문 내 등장 순서(0-base)
  raw: string;     // 원본 토큰 문자열
}

export const make = (type: MarkerType, key = '0') => `[[PF:${type}:${key}]]`;

export function scan(body: string): Marker[] { /* MARKER_RE로 순서대로 수집 */ }
export function strip(body: string): string { /* 모든 마커 제거(미리보기용) */ }
```

> 생성기·비주얼·합성기·검증기는 마커 문자열을 **직접 쓰지 않고** `make()`/`scan()`만 사용한다.

## 4. 프롬프트 규약

### 4.1 프롬프트 조립 순서 (③ 본문 생성기)

```
[시스템 지시(고정)]  +  [사용자 프롬프트(주제별)]  +  [주제 키워드]  +  (선택)[참고 자료]
```

- 사용자 프롬프트는 **이름을 붙여 저장/불러오기/삭제**한다(R-2.1).
- 참고 자료는 "제목+본문 가져오기"로 수집한 현재 페이지 본문(선택).

### 4.2 시스템 지시 템플릿 (고정 부분, 요지)

AI가 항상 지키도록 강제할 항목:

1. 한국어 네이버 블로그 본문을 작성한다.
2. 소제목은 `##`(H2)로 구분한다.
3. 삽입 지점에는 **지정된 마커만** 정확히 출력한다(임의 마커 생성 금지).
   - 각 H2 바로 아래 `[[PF:H2THUMB:n]]` (n = 소제목 순번)
   - 본문 이미지가 필요한 곳에 `[[PF:IMG:n]]`
   - (요청 시) `[[PF:AD:0]]`, `[[PF:PRODUCT:...]]` 등
4. **출력은 본문 텍스트만** — 코드펜스(```), "다음은 글입니다" 같은 설명·머리말·꼬리말 금지.
5. 마커는 한 줄에 단독으로 둔다.

> 마커 표기·종류는 본 문서 2장에서 그대로 가져와 시스템 지시에 박는다(여기서도 단일 출처 원칙 유지: 지시문 생성 시 `markers.ts`를 참조해 예시를 만든다).

### 4.3 생성 방식 (R-2.3)

- **직접 호출**: 인증 키로 AI 직접 호출. 키 할당량 초과 시 다음 키로 자동 전환.
- **대화형 자동화**: AI 서비스 화면 자동 조작(키 불필요, 대기시간 설정).
- 실패 시: 재시도 → 키 전환 → 그래도 실패면 사유 통지.

## 5. 출력 후처리 (③ → 본문 확정)

생성 직후 순서대로 적용한다.

1. **래퍼 제거**(R-2.2, 8.2-3): 앞뒤 코드펜스(` ``` `, ` ```html `), "본문:" 같은 설명 머리말/꼬리말 제거.
2. **마커 검사**(R-2.2): `scan()`으로 마커를 수집해 **약속한 마커가 들어있는지** 검증. 없으면 재생성 후보.
3. **구조 검사**: `<h2>`(또는 `##`) 개수를 센다 → 5장 정합성 검사의 기준값.
4. 통과한 본문만 다음 단계(비주얼·합성)로 넘긴다.

---

## 6. 정합성 검사 (⑨·④ 진입 전 — 합성 차단 조건)

| 검사 | 규칙 | 위반 시 |
|---|---|---|
| H2 ↔ H2THUMB | `H2 개수` == `H2THUMB 마커 수` (R-7.3) | 합성 차단 + 통지 |
| 이미지 마커 ↔ Visual | `IMG/H2THUMB/H1THUMB 마커 수` == `생성 Visual 수` (R-7.6, 16장 계약2) | 합성 차단 + 통지 |
| 마커 순서 | 본문 마커 순서 == 삽입 큐 순서 (R-3.2) | 재배치 금지(에러) |
| 광고문구 | 옵션 ON이면 `AD` 마커 ≥ 1 존재 (R-3.4) | 합성 차단 + 통지 |

> 합성기(④)는 `scan()` 결과 순서대로 **삽입 작업 큐(InsertQueue)** 를 만든다(R-3.3). 이 큐가 07 문서(에디터 자동화)의 입력이 된다.

---

## 7. 변환·편집 시 마커 보존 (⑩ 검증·편집기)

- HTML↔Markdown 변환, 찾기·바꾸기 후에도 마커는 **그대로 남아야** 한다(R-8.3, R-8.4).
- 플레인 토큰(`[[PF:...]]`)이라 turndown/marked를 왕복해도 일반 텍스트로 보존된다. **DoD: HTML→MD→HTML 왕복 후 `scan()` 개수 불변.**
- 미리보기에서는 `strip()`으로 마커를 숨겨 렌더한다(원본은 보존).

---

## 8. 합의가 필요한 결정 (열린 항목)

- `AD` 문구의 기본 위치(첫 H2 위 / 본문 맨 끝) — 정책·가독성 절충 필요.
- `PRODUCT`/`SHOP` 키 네이밍 규칙(상품 ID 직접 vs 내부 키 매핑).
- 마커를 한 줄 단독이 아니라 **문장 중간**에 허용할지(현재는 블록 기준으로 단독 권장).

> 위 항목이 정해지면 본 문서와 `markers.ts`를 함께 갱신한다(단일 출처 유지).
