# 펜플로우 (PenFlow) — 네이버 에디터 자동화 명세서

> 기준 문서: `02-functional-design.md`(12장, R-4.x, R-5.x, 17장), `03-implementation-checklist.md`(7·10장), `04-tech-stack.md`(⑥ 삽입 엔진, `world:MAIN`), `06-marker-prompt-spec.md`(삽입 큐 입력)
> 목적: 펜플로우의 **심장**인 삽입 엔진(⑥)이 네이버 SmartEditor를 어떻게 조작하는지 **셀렉터·기법·순서·타임아웃 수준**으로 확정한다.

> ⚠️ **최고 위험 영역.** 네이버 에디터는 우리가 제어하지 않는 외부 화면이고 UI가 바뀔 수 있다. **코딩 전에 짧은 기술 검증(스파이크 — 본 문서 §12 체크리스트)으로 미지수를 먼저 확정**할 것. 이 문서는 스파이크 결과를 채워 넣을 골격이기도 하다.

---

## 1. 실행 위치와 환경

- **content script**가 네이버 글쓰기 페이지에 주입되어 동작한다.
- SmartEditor는 본문 영역이 별도 컨텍스트(iframe/격리)일 수 있어, 에디터 내부 API·이벤트 접근이 필요하면 **`world: "MAIN"` 주입**을 사용한다(04). 격리 월드만으로 가능한지 여부는 **스파이크에서 확정**.
- 입력은 가능하면 **사용자 동작을 모사**(클립보드 붙여넣기·클릭·키 이벤트)해 안정성을 높인다.

---

## 2. 전체 동작 순서 (Happy Path)

```
1) 편집기 준비 감지 (폴링 + 타임아웃)        ── R-4.1
2) 초기 팝업·배너 정리
3) 본문 영역 포커스 확보                      ── R-4.2
4) 제목 입력
5) 삽입 작업 큐(InsertQueue) 순차 실행        ── R-3.3 / 06 6장
     텍스트·HTML → 이미지(+링크) → 표/백링크 → 쇼핑링크 → 광고문구 …
6) 본문 전체 선택 → 서식 일괄 적용             ── 12.2-5
7) 썸네일 이미지에 링크 부착
8) 완료 신호 발신 → ⑦ 발행 처리기 / ⑧ 오케스트레이터  ── R-5.2
```

> 큐의 **순서는 06의 마커 순서를 그대로 따른다**(R-3.2). 엔진은 재배치하지 않는다.

---

## 3. 셀렉터 전략 (R-4.1)

UI 변경에 대비해 **셀렉터를 코드에 흩지 말고 상수 파일 1개**(`selectors.ts`)에 모으고, 각 타깃은 **폴백 배열**로 둔다.

```ts
// 우선순위 순서대로 시도. 앞에서 실패하면 다음으로 폴백.
export const SEL = {
  editorReady: ['.se-editing-area', '[data-editor-ready]', 'iframe.editor'],
  titleField:  ['.se-title-text', '[placeholder*="제목"]'],
  bodyArea:    ['.se-content', '.se-editing-area [contenteditable="true"]'],
  initialPopupClose: ['.se-popup-close', '.btn_close'],
  saveDraft:   ['button[data-name="save"]', '.save_btn'],
  publishOpen: ['.publish_btn', 'button[data-name="publish"]'],
  publishConfirm: ['.confirm_btn', 'button[data-testid="seOnePublishBtn"]'],
} as const;
```

> 위 셀렉터 값은 **예시**다. 실제 값은 스파이크로 채운다. 셀렉터가 모두 실패하면 "UI 변경 의심" 오류로 처리(8장).

---

## 4. 편집기 준비 감지 (R-4.1)

- `SEL.editorReady`를 **폴링**(예: 200~400ms 간격)하며 `timeout`(예: 10s)까지 대기.
- 준비 조건: 본문 contenteditable이 존재하고 입력 가능 상태.
- 타임아웃 시: "편집기 영역 미발견" 오류 + 사용자 안내(17장: 타임아웃 + 안내). 패닉 금지.
- **초기 팝업 정리**: `SEL.initialPopupClose`가 있으면 닫는다(저장 안내·도움말 등).

---

## 5. 포커스 처리 (R-4.2)

- 본문 입력 전 `SEL.bodyArea` 클릭으로 포커스를 준다.
- 포커스 확보 실패(입력이 안 들어감) 시: **"본문을 한 번 클릭하세요" 안내 + 재시도**. 자동 재시도 횟수 제한 후에도 실패면 안전 종료.

---

## 6. 삽입 작업 큐 (InsertQueue)

06의 `scan()` 결과로 만든 큐를 **순차** 실행한다. 작업 타입과 주입 기법:

| Job | 입력 | 주입 기법(권장) | 비고 |
|---|---|---|---|
| `TEXT`/`HTML` | 본문 조각 | 클립보드 `DataTransfer` 붙여넣기 또는 contenteditable 직접 삽입 | 서식 유지 위해 paste 우선 |
| `IMAGE` | 이미지 + (선택)링크 | 에디터 이미지 삽입 경로(업로드/URL) | 업로드 **완료 비동기 대기** 필수 |
| `H2THUMB`/`H1THUMB` | 썸네일 이미지 | IMAGE와 동일 + 링크 부착 | 링크는 8단계에서 부착 |
| `PRODUCT` | 상품표 | HTML 표 삽입 | 최신 데이터 반영(R-3.5) |
| `SHOP` | 제휴 링크 | 링크 삽입 후 **불필요 텍스트 제거** | |
| `BACKLINK` | 백링크 블록 | HTML 삽입 | |
| `CTA` | 버튼 | HTML 삽입 | |
| `AD` | 광고/협찬 문구 | 텍스트 삽입 | 옵션 ON이면 누락 금지(R-3.4) |

- **작업 간 대기시간**은 설정값으로 조절(하드코딩 금지, R-4.3). 이미지 업로드처럼 비동기 완료가 필요한 작업은 **완료 감지 후** 다음으로 진행.
- 마커에 대응 리소스가 없으면 스킵(단, `AD`는 예외, R-3.1).

---

## 7. 제목 입력 / 서식 적용

- **제목**: 본문에서 제목을 추출(또는 주제에서 결정)해 `SEL.titleField`에 입력.
- **서식 일괄 적용**(12.2-5): 본문 전체 선택 → 줄간격·글꼴·글자크기 적용. 적용 값은 설정에서 가져온다.
- **썸네일 링크 부착**: 대표/소제목 썸네일 이미지에 지정 링크를 건다.

---

## 8. 오류 처리 / 중단 (R-4.4, 17장)

- 한 작업이 실패해도 **패닉 없이**: 실패한 작업을 **명시**(어떤 마커/Job인지)하고 안전 종료 또는 사용자 선택(건너뛰고 계속 / 중단).
- **삽입 중 중단** 요청 시: 진행 중 작업 정리 후 종료.
- 셀렉터 전부 실패: "UI 변경 의심" 오류로 통지(폴백 셀렉터 갱신 필요 신호).

---

## 9. 편집기 모드 안내 (R-4.5)

- 사이드바 모드에서는 에디터 감지가 실패할 수 있으므로 **팝업 모드 사용을 안내**한다.
- 모드 감지 가능 시 부적합 모드면 사전 경고.

---

## 10. 완료 신호 (R-5.2 → ⑦/⑧)

- 삽입이 끝나면 **"삽입 완료"** 신호를 발신한다.
- 발행 처리기(⑦)가 임시저장/발행을 수행하고, 그 결과가 오케스트레이터(⑧)로 전달되어 **다음 글 전이**의 트리거가 된다(중복 발행 방지: 완료 신호 수신 후에만 전이, R-6.2).

---

## 11. 타임아웃·대기시간 파라미터 (전부 설정값, 하드코딩 금지 — R-4.3)

| 파라미터 | 권장 초기값 | 설명 |
|---|---|---|
| `editorReadyPollInterval` | 300ms | 준비 감지 폴링 간격 |
| `editorReadyTimeout` | 10s | 준비 감지 최대 대기 |
| `jobDelayMin~Max` | 300~900ms | 작업 간 대기(무작위 권장) |
| `imageUploadTimeout` | 30s | 이미지 업로드 완료 대기 |
| `focusRetry` | 3회 | 포커스 재시도 횟수 |

## 12. 스파이크 체크리스트 (코딩 전 검증할 미지수)

- [ ] 본문이 iframe인가, 같은 문서 내 contenteditable인가? 격리 월드에서 접근 가능한가, `world:MAIN` 필요한가?
- [ ] 텍스트/HTML 주입은 클립보드 paste가 먹히는가? 붙여넣기 시 서식 sanitize는 어떻게 되는가?
- [ ] 이미지 삽입 경로(업로드 vs URL)와 **업로드 완료를 감지할 신호**(DOM 변화/네트워크)
- [ ] 제목 필드·저장·발행·확인 버튼의 **안정적인 셀렉터**(동적 클래스 회피)
- [ ] 서식(줄간격·글꼴·크기) 적용 UI의 조작 방법
- [ ] 초기 팝업의 종류와 닫기 셀렉터
- [ ] 팝업/사이드바 모드 차이가 위 모든 항목에 미치는 영향

> 이 체크리스트가 다 채워지면 3장 셀렉터·6장 기법·11장 파라미터의 "예시" 값을 **실측값으로 교체**하고, M1 삽입 엔진 구현에 착수한다.

---

## 13. 알려진 함정

- **동적 클래스명**: SmartEditor의 일부 클래스는 빌드마다 바뀔 수 있어, 의미 기반 셀렉터·폴백·DOM 구조 기준 탐색을 병행한다.
- **isolated world 경계**: content script 기본 월드에서 에디터 내부 JS 상태에 접근 못 할 수 있다 → `world:MAIN` 또는 이벤트 모사.
- **비동기 업로드**: 이미지가 화면에 "보이는" 시점과 에디터에 "확정"되는 시점이 다르다 → 완료 신호를 기다리지 않으면 다음 작업과 충돌.
- **붙여넣기 정제**: 에디터가 붙여넣은 HTML을 자체 정제하면서 일부 마크업/마커가 사라질 수 있다 → 06의 플레인 토큰 마커가 유리하지만, 삽입 단계에서는 이미 리소스로 치환되어야 함(마커가 본문에 남으면 안 됨).
